## _The person behind the tests..._

Born a musician, grown as a gardener, refined through design, and now methodical as a tester. The thread connecting all of it has always been the same: a sensitivity to when something is off.

A bootcamp showed me what I hadn't seen before: QA and Design share the same core mindset. Both are about noticing what's wrong before anyone else does. That realization changed everything. I stopped designing interfaces and started breaking them instead.

Today I work across the full testing stack — manual testing, test case design, API validation, SQL, UI automation with Selenium and Pytest, and Android testing. My approach is a mix of instinct and structure: I follow the process, but I also chase the edge case that the process didn't think to cover.

Right now I'm focused on landing a role in a product company where I can grow alongside a solid team, deepen my testing and automation skills, and eventually own the full testing layer of a product. 

*Also learning patience. Customer service is doing its part.*

*Always testing. Always improving...*

### Manual Chaos — Testing a Store Built to Break _(Saucedemo.com)_

![Manual Testing](https://img.shields.io/badge/Manual_Testing-6B7280?style=flat-square)
![Chrome DevTools](https://img.shields.io/badge/Chrome_DevTools-4285F4?style=flat-square&logo=googlechrome&logoColor=white)
![Google Sheets](https://img.shields.io/badge/Google_Sheets-34A853?style=flat-square&logo=googlesheets&logoColor=white)

**_The challenge_**

Saucedemo is a sandbox e-commerce app built by Sauce Labs, designed to simulate a complete purchase flow with six intentionally broken user profiles. The goal was to design and execute a full manual QA suite — not just run through the happy path, but systematically expose what breaks, how it breaks, and why it matters.

**_The process_**

Test cases were designed before a single test was executed. UX/UI and functional cases were kept separate from the start, with a deliberate decision to use a single set of cases executed across all six users — rather than designing six separate suites — to demonstrate reusability and equivalence partitioning. A compatibility matrix was built based on real usage statistics from StatCounter, and Chrome DevTools was used throughout execution to go beyond surface-level bug reporting: DOM inspection, Network tab analysis, and Event Listener review were used to identify root causes and rule out backend vs. frontend origins.

**_The results_**

- 116 test cases · 6 users · 3 platform/browser combinations
- 32 bugs documented (BUGSL-001 to BUGSL-032) — 2 Critical, 13 High, 8 Medium, 9 Low
- Key findings: a cross-user cart privacy violation, a backend 503 crash triggered by an invalid product entry, a completely broken checkout flow in two user profiles, and a visual user riddled with misplaced elements and incorrect prices
- Key lesson: a well-designed test suite doesn't just find bugs — it tells a story about where the product is solid and where it falls apart

📁 [View Repository](https://github.com/alejandrog-gutierrez/Saucedemo_test_suite_manual)

### Breaking a Fake Store — API Automation Suite _(Fakestoreapi.com)_

![Python](https://img.shields.io/badge/Python-3776AB?style=flat-square&logo=python&logoColor=white)
![Pytest](https://img.shields.io/badge/Pytest-0A9EDC?style=flat-square&logo=pytest&logoColor=white)
![Requests](https://img.shields.io/badge/Requests-3776AB?style=flat-square&logo=python&logoColor=white)
![Postman](https://img.shields.io/badge/Postman-FF6C37?style=flat-square&logo=postman&logoColor=white)

**_The challenge_**

The goal wasn't just to confirm the API worked — it was to find where the contract broke. FakeStore simulates an e-commerce platform with auth, users, products, and carts. The real question was: how does it behave when you feed it things it wasn't designed for?

**_The process_**

Each endpoint was first explored manually in Postman to understand its behavior before writing a single automated test. From there, a modular suite was built with a dedicated layer for HTTP requests, test data, and reusable assertions — keeping test files clean and focused on scenarios, not plumbing. Coverage included functional, negative, boundary, and edge case scenarios across all four endpoints.

**_The results_**

- 198 tests · 81 passed · 117 documented expected failures · 0 errors
- 1 critical bug — a plain string payload to `/auth/login` crashed the server with a 524 timeout
- 8 major findings — including DELETE returning full objects instead of empty bodies, and POST accepting completely invalid payloads with 201
- Key lesson: an API that never returns an error isn't reliable — it's just quiet about being broken

📁 [View Repository](https://github.com/alejandrog-gutierrez/FakestoreAPI_automated_testing_suite)

### Booking a Ride, Automatically — UI End-to-End Suite _(Urban Routes)_

![Python](https://img.shields.io/badge/Python-3776AB?style=flat-square&logo=python&logoColor=white)
![Pytest](https://img.shields.io/badge/Pytest-0A9EDC?style=flat-square&logo=pytest&logoColor=white)
![Selenium](https://img.shields.io/badge/Selenium-43B02A?style=flat-square&logo=selenium&logoColor=white)

**_The challenge_**

Automate a complete taxi booking flow in a dynamic web app — from entering a route to confirming driver assignment — without the suite breaking every time the UI shifted.

**_The process_**

The focus was deliberately on the happy path: the business-critical sequence that, if broken, prevents any user from completing a booking. The trickiest part wasn't writing the tests — it was making them stable. Payment couldn't be added without prior phone verification, and the "link" button only activated after the CVV field lost focus. That kind of UI dependency required explicit waits and simulating real user behavior, not just clicking elements. The Page Object Model kept locators and actions separated from test logic.

**_The results_**

- Full end-to-end flow automated across 9 dependent steps
- Solved UI synchronization issues that caused flaky test execution
- Key lesson: dynamic UIs don't just test your automation skills — they test your patience and your locator strategy

📁 [View Repository](https://github.com/alejandrog-gutierrez/Project_Urban_Routes)
